About TradeLab's Storage

Project Background

TradeLab started as a class project but evolved into something much more meaningful. As a junior software engineer who also trades actively, I noticed a significant gap in the market: while there are excellent trading journal solutions available, many come with substantial subscription costs that can be challenging for beginners.

Having just completed my coding classes, I decided to transform my project into a real-world application that could help traders like myself. The goal was clear: create a comprehensive, powerful trading journal that remains accessible to everyone, especially those just starting their trading journey.

Project Vision: As both a developer and trader, I understand the importance of having robust tools without breaking the bank. TradeLab aims to provide professional-grade features while keeping costs minimal through smart technical choices.

Understanding Browser Storage

Modern web browsers offer several ways to store data locally. Let's explore each option and understand why we chose IndexedDB for TradeLab:

Local Storage

  • What it is: Simple key-value storage mechanism in the browser
  • Storage Limit: Usually 5-10 MB per domain
  • Data Persistence: Persists until explicitly cleared, survives browser restarts
  • Security: Vulnerable to XSS attacks, data stored as plain text
  • Why Not for TradeLab:
    • Limited storage capacity for trade data
    • No structured query support
    • Can't store complex data types (only strings)
    • No built-in indexing for performance

Session Storage

  • What it is: Similar to Local Storage but with session-based persistence
  • Storage Limit: Also typically 5-10 MB
  • Data Persistence: Only lasts for one browser session, cleared when tab closes
  • Security: Slightly more secure than Local Storage due to session isolation
  • Why Not for TradeLab:
    • Temporary nature unsuitable for trade records
    • Same limitations as Local Storage
    • Data loss risk when closing browser

Cache API

  • What it is: Modern API for storing network requests and responses
  • Storage Limit: Based on available disk space
  • Data Persistence: Managed by service workers, survives browser restarts
  • Security: Protected by same-origin policy, requires HTTPS
  • Why Not for TradeLab:
    • Designed for caching network resources
    • Not optimized for structured data storage
    • Lacks querying and indexing capabilities

IndexedDB (Our Choice)

  • What it is: Low-level API for client-side storage of structured data
  • Storage Limit: Generally >50MB, can extend with user permission
  • Data Persistence: Permanent until explicitly deleted, survives browser restarts
  • Security Features:
    • Protected by same-origin policy
    • Built-in transaction support
    • Data integrity protection
    • Resistant to XSS attacks
  • Perfect for TradeLab Because:
    • Supports complex data structures needed for trades
    • Provides robust querying capabilities
    • Offers excellent performance with indexes
    • Handles large datasets efficiently
    • Built-in data integrity protection

A Note on Security

While browser storage options have different security characteristics, it's important to understand that:

  • All browser storage is isolated per domain (same-origin policy)
  • No browser storage is suitable for sensitive data like passwords
  • Data should be encrypted if storing sensitive information
  • Regular backups are essential regardless of storage method

Data Protection: TradeLab implements additional security measures beyond the basic browser storage security:

  • Data validation before storage
  • Regular integrity checks
  • Automated backup reminders
  • Export functionality for manual backups

For more detailed information about browser storage, check out these resources:

  • Mozilla's IndexedDB Documentation
  • Web Storage Overview

    • Why IndexedDB?

    We chose IndexedDB as our primary storage solution for several compelling reasons:

    • Database-Like Capabilities:
      • Complex data structures support
      • Efficient indexing and querying
      • Transaction support for data integrity
      • Significant storage capacity (generally >50MB)
    • Cost Benefits:
      • No server costs passed to users
      • No need for expensive database hosting
      • Completely free for users
    • Privacy:
      • Data stays on your device
      • No external data transmission
      • Complete control over your information

    Storage Capacity Analysis

    Let's break down the storage requirements for trade data:

    Single Trade Object (Approximate Size):

    • Date: 8 bytes
    • Symbol: ~10 bytes
    • Entry/Exit Prices: 16 bytes
    • Position Size: 8 bytes
    • Notes (average): 200 bytes
    • Additional Metadata: 50 bytes
    • Total per trade: ~300 bytes

    Storage Calculations:

    • Active trader (20 trades/day): 6,000 bytes/day
    • Monthly usage: ~180KB
    • Yearly usage: ~2.2MB

    Supabase Free Tier (1GB):

    • Can support: ~500 active traders
    • Or: ~2,000 moderate traders
    • Or: ~5,000 casual traders
